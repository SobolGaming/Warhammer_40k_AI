# Contract 2 to Contract 3 migration

Contract 3.0 makes Phase 18I interaction metadata part of the closed public
contract. It is intentionally not wire-compatible with Contract 2.

Clients must regenerate models from Contract 3 OpenAPI and schemas, then:

1. require `decision-request-view-v2-interaction`,
   `interaction-descriptor-v2-variants`, `game-view-v6-interaction`, and
   `session-projection-v2-interaction`;
2. render from `interaction_kind` and `submission_variants`, using nullable
   selection cardinality as “not statically known” rather than assuming one;
3. construct finite or parameterized wrappers from `submission_schema_ref`,
   and validate parameterized bodies against the selected variant's
   `proposal_schema_ref`;
4. consume `nested_interaction_requests` as typed annotated requests and apply
   the parent pending decision's viewer visibility;
5. discard Contract 2 projection hashes and fetch a fresh Contract 3
   projection, because the hash now covers the complete viewer-visible view;
6. use a separately deployed 2.x adapter for clients that cannot migrate
   during the published support window.

There is no content negotiation between majors. A request with a mismatched
payload `schema_version` fails before engine mutation.
